# ADR-001 Источник данных

Решение:

Используем только MOEX ISS API.

Причины:

бесплатно;
не зависит от брокера;
можно запускать где угодно;
достаточно для MVP.

# ADR-002 Частота сканирования

Решение:

Запуск один раз в день.

20:00 МСК

Причины:

стратегия не внутридневная;
облигации меняются медленно;
меньше нагрузки на API;
проще эксплуатация.

# ADR-004 История цен

Решение:

Сохранять ежедневный снимок рынка.

Причины:

анализ трендов;
ретроспективное тестирование;
обучение будущих моделей.

# ADR-005 НКД обязателен

Решение:

Прибыль считается только с учетом НКД.

Теперь самое важное

Давай разберём НКД на примере.

Представим:

Номинал: 1000

Цена на бирже: 980

Купон через 5 дней: 40

Погашение через 15 дней

Новичок считает:
1000 + 40 - 980 = +60 руб
Выглядит шикарно.

Но допустим:

Но допустим:

НКД = 35 рублей

Фактически покупатель платит:

980 + 35 = 1015

Тогда:

1000 + 40 - 1015 = 25 рублей

Доходность упала более чем в два раза.
Поэтому формула стратегии должна быть такой:

Expected Profit

=
Nominal

+
Remaining Coupons

-

Market Price

-

NKD

-

Broker Fee

Именно эта формула станет основой Bond Scout.


# Bond Scout Architecture

## Overview

Bond Scout состоит из следующих компонентов:

### Scanner

Получение данных о рынке облигаций из MOEX ISS API.

Ответственность:

* загрузка списка облигаций;
* загрузка цен;
* загрузка НКД;
* загрузка информации о купонах.

---

### Database Layer

Хранение:

* облигаций;
* истории цен;
* результатов сканирования;
* результатов анализа.

Используется PostgreSQL.

---

### Strategy Engine

Расчёт:

* прибыли до погашения;
* доходности;
* рейтинга кандидатов.

---

### AI Analyst

Дополнительный анализ эмитентов.

Функции:

* анализ новостей;
* анализ риска дефолта;
* формирование краткого заключения.

На MVP не используется.

---

### Telegram Module

Отправка ежедневных отчетов.

---

### API Layer

REST API для:

* просмотра облигаций;
* просмотра кандидатов;
* запуска сканирования.

---
